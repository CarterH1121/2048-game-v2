## 已知风险与后续建议

1. 后端仍采用“昵称登录/注册”契约，不能等同于完整认证。需要与 `2048-admin` 的真实单体后端单独完成契约和认证治理。
2. 单文件 SPA 接近万行，维护风险高；后续可先做无行为变化的模块抽取方案和快照回归，不能直接重写。
3. 大量内联样式和脚本使严格 CSP 难以落地。本轮已封堵已确认的服务端文本注入点，但完整安全治理仍需系统性 DOM sink 审计。
4. Google Fonts 是跨域可选资源；离线时使用系统字体，视觉会有轻微差异。
5. 历史根脚本存在“发现失败仍输出审核通过”以及旧模块名断言。`npm test` 已不依赖这些结论；保留它们仅供历史对照。
6. 接管验收早期在拦截规则启用前，浏览器曾向当时配置的生产 API 发出一次昵称登录请求。未执行任何数据库回滚或其他生产写操作；如需确认影响，应由获授权人员检查后端用户/访问日志。
7. 首次断网创建的游客档案不会自动升级为云端账号；当前优先保证本地可玩且不误上报，后续需要产品定义“游客绑定/迁移”流程。
